Chapter 5: Names, Bindings, and Scopes

  • Learning objectives:

Names

  • Design Issues:

    • Case sensitivity

    • Special words (reserved vs keywords)

  • Name Forms:

    • Usually a letter followed by letters, digits, underscores

    • Camel notation in C-based languages (e.g., myStack)

    • Special characters in some languages:

      • PHP: $ prefix required

      • Perl: $, @, % prefixes indicate type

      • Ruby: @ and @@ indicate instance/class variables

  • Case Sensitivity:

    • C-based languages are case sensitive

    • Example: rose, ROSE, and Rose are distinct

    • Tradeoff between flexibility and readability

Variables

  • Variable Attributes (Sextuple):

    • Name

    • Address (l-value)

    • Value (r-value)

    • Type

    • Lifetime

    • Scope

  • Address:

    • Machine memory location

    • Can change during execution

    • Multiple variables can share address (aliasing)

  • Aliases:

    • Multiple variables referencing same memory

    • Created through:

      • Union types

      • Pointer variables

      • Reference variables

      • Parameter passing

Type Bindings

  • Static Type Binding:

    • Type fixed at compile time

    • Through explicit/implicit declarations

    • Examples: Java, C++, C#

  • Dynamic Type Binding:

    • Type determined at runtime

    • Changes possible during execution

    • Examples: Python, JavaScript

    • Advantages:

      • More flexible

      • Supports generic code

    • Disadvantages:

      • Less reliable

      • Runtime overhead

      • Harder to optimize

Variable Categories by Lifetime

  • Static Variables:

    • Bound to memory before execution

    • Retain value between calls

    • Examples: global variables, static locals

  • Stack-Dynamic Variables:

    • Created upon declaration elaboration

    • Deallocated upon block exit

    • Used for local variables in functions

  • Explicit Heap-Dynamic Variables:

    • Allocated/deallocated by explicit commands

    • Accessed through pointers/references

    • Example: new/delete in C++

  • Implicit Heap-Dynamic Variables:

    • Bound to heap storage on assignment

    • All attributes bound at assignment

    • Example: JavaScript arrays

Scope

  • Definition: Range where variable is visible

  • Categories:

    • Local variables (declared in unit)

    • Nonlocal variables (visible but not local)

    • Global variables (special nonlocal case)

Static Scope

  • Based on program structure

  • Rules:

    • Search starts in local unit

    • Continues in enclosing units

    • Follows static nesting

  • Advantages:

    • Determinable at compile time

    • Better readability

    • More efficient

Dynamic Scope

  • Based on call sequence

  • Rules:

    • Search starts in local unit

    • Continues through calling sequence

    • Follows dynamic calling chain

  • Characteristics:

    • More flexible

    • Harder to read/understand

    • Less efficient

    • Runtime binding resolution

Blocks and Declaration Order

  • Blocks:

    • Create new static scopes

    • Allow local variables

    • Stack-dynamic storage

    • Example: compound statements in C/C++

  • Declaration Order:

    • Some languages require declarations first

    • Others allow mixed declarations/statements

    • Affects variable visibility

    • Scope can start at declaration or block start

Named Constants

  • Purpose:

    • Improve readability

    • Aid program maintenance

    • Parameterize programs

  • Implementation:

    • C++: const keyword

    • Java: final keyword

    • C#: const and readonly

Summary

  • Variables characterized by six attributes

  • Binding links attributes to entities

  • Four categories of variable lifetime

  • Static vs dynamic scoping tradeoffs

  • Blocks create local scopes

  • Named constants improve maintainability
